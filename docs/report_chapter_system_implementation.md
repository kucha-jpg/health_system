# 第X章 系统实现与测试

## 1. 系统实现概述

本课题围绕院外健康管理场景，采用前后端分离的 B/S 架构完成系统实现。后端基于 Spring Boot 构建 RESTful API，承担身份认证、权限控制、业务编排与数据访问；前端基于 Vue 3 构建多角色交互界面，承担信息展示、交互操作与流程引导；MySQL 负责业务数据持久化，Flyway 用于数据库版本迁移管理。

系统实现目标聚焦于三类核心角色：患者、医生与管理员。患者侧强调数据持续上报与自我管理，医生侧强调群组协作与风险干预，管理员侧强调平台治理与运行监控。通过角色分工与流程闭环，系统实现了“数据采集-风险识别-干预处理-运营治理”的持续管理链路。

## 2. 技术路线与架构落地

### 2.1 前后端分离架构

系统采用前后端解耦模式。前端通过统一 HTTP 模块调用后端接口，后端以控制器-服务-数据访问分层组织业务逻辑。该模式降低了模块耦合度，便于后续功能扩展与独立部署。

### 2.2 安全与权限机制

系统安全能力由 Spring Security 与 JWT 共同实现：

1. 登录成功后签发 JWT，前端在会话中携带令牌访问受保护资源。
2. 后端过滤器解析令牌并恢复用户身份与角色信息。
3. 依据 URL 前缀进行角色隔离：管理员、医生、患者仅可访问各自权限范围内接口。
4. 通过登录版本号机制实现同账号互斥登录，保障会话一致性与安全性。

该机制满足了医疗场景下对鉴权准确性、会话时效性和越权防护的基本要求。

### 2.3 数据模型与迁移策略

系统采用关系型数据建模，核心数据实体包含：用户、角色、患者档案、健康数据、健康预警、医生群组、系统公告、操作日志、反馈消息、预警规则等。通过 Flyway 进行版本化迁移，保证不同环境下数据库结构一致、演进可追踪。

## 3. 功能模块实现

### 3.1 患者端功能实现

患者端重点实现如下能力：

1. 账户注册与登录。
2. 个人档案维护（基础信息、病史、用药史、过敏史等）。
3. 健康数据上报与历史查询（支持血压、血糖、体重、服药等指标）。
4. 预警详情查看（展示预警等级、原因与状态）。
5. 周报/月报汇总（统计上报量、预警量与指标分布）。

在数据质量控制方面，系统对不同指标设置格式与取值校验规则，避免异常输入导致分析偏差。

### 3.2 医生端功能实现

医生端围绕“分组管理+风险干预”实现业务协同：

1. 创建和维护医生群组。
2. 将患者加入指定群组并查看成员列表。
3. 查看系统触发的预警并执行闭环处理。
4. 查看群组内患者档案与连续趋势洞察（含近期预警记录）。

其中，患者洞察能力用于支撑医生对重点对象进行纵向跟踪，增强干预决策的及时性与针对性。

### 3.3 管理员端功能实现

管理员端覆盖平台治理与运营分析能力：

1. 用户管理：患者/医生账号增删改查、启停控制。
2. 角色权限管理：角色权限字符串配置与更新。
3. 基础数据管理：系统公告管理、预警规则管理。
4. 系统监控与日志：监控概览、日志分页查询与导出。
5. 反馈治理：用户反馈处理、回复、统计分析与批量操作。

新增的预警规则管理模块实现了阈值后台化配置，预警引擎优先读取数据库规则，提升了业务策略可维护性。

## 4. 预警机制实现说明

系统预警逻辑采用“数据上报触发评估”的方式：患者每次上报后，系统根据指标类型进行规则判定并生成风险级别。当前支持的规则类型包括血压、血糖、体重。为解决阈值固化问题，系统已引入预警规则配置表，管理员可在线维护高风险/中风险阈值及启用状态。若未匹配到启用规则，系统回退至默认阈值策略，保证业务连续性。

该实现兼顾了两方面要求：

1. 规则可配置：满足平台运营中对阈值动态调整的需求。
2. 判定可兜底：避免规则缺失导致预警能力失效。

## 6. 小结

综合实现与测试结果可见，本系统已完成课题目标中的主要功能落地，并在安全控制、规则配置化、医生协作洞察和管理员治理能力方面形成较完整的实践方案。系统能够支撑院外健康管理的连续数据追踪与主动预警干预，为医疗团队开展分组化、精细化管理提供了可用技术基础。

3. 在日志与反馈数据基础上建设更完善的运营分析指标体系。
