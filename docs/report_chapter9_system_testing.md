# 第9章 系统测试

为验证系统在真实使用场景下的正确性、稳定性与可维护性，本章从测试用例设计、功能测试、性能测试和结果分析四个方面进行说明。测试覆盖患者、医生、管理员三类角色的关键业务链路，同时结合自动化脚本、后端单元测试与集成测试形成多层验证体系。

## 6.1 测试用例设计

### 6.1.1 设计目标

测试用例设计围绕以下目标展开：

1. 验证核心业务闭环可用：注册、登录、上报、预警、处理、反馈。
2. 验证权限边界可靠：未登录拦截、角色隔离、医生团队数据隔离。
3. 验证规则联动正确：指标启停、个性化阈值、趋势预警、连续高风险判定。
4. 验证系统可追溯：操作日志与错误码输出满足问题定位需求。

### 6.1.2 用例覆盖范围

本系统已设计并沉淀 17 个核心场景，覆盖以下测试域：

1. 认证与会话：注册登录、同账号互斥登录、多账号并行登录。
2. 患者业务：健康上报、历史查询、预警查看、周报月报、阈值配置。
3. 医生业务：预警筛选与处理、群组管理、患者洞察、团队隔离。
4. 管理员业务：角色权限、预警规则、指标类型配置、反馈治理。
5. 错误码链路：400、401、403、404、409 统一断言。
6. 展示与交互：图表渲染、筛选联动、页面可用性。

### 6.1.3 测试方法与分层

测试采用“自动化为主、手工为辅”的分层策略：

1. 接口回归测试：通过脚本执行端到端场景，验证业务链路。
2. 错误码门禁测试：独立断言高频错误码，防止安全链路回归。
3. 后端单元测试：验证服务层核心规则与异常分支。
4. 后端集成测试：验证迁移与安全链路在真实依赖下的行为。
5. 前端手工测试：验证角色页面流程、状态反馈和视觉一致性。

### 6.1.4 测试环境

1. 操作系统：Windows（支持 PowerShell 脚本执行）。
2. 运行方式：Docker 组合启动 mysql、backend、frontend。
3. 访问地址：后端接口基址为 http://127.0.0.1:9090/api。
4. 数据基础：数据库初始化后，使用脚本动态生成隔离测试账号与测试标记。

## 6.2 功能测试

### 6.2.1 核心功能场景执行

功能测试重点覆盖以下业务链路：

1. 患者注册登录后上报血压并查询历史数据。
2. 患者上报异常指标后医生可在工作台收到预警。
3. 医生可创建群组并维护患者成员。
4. 管理员可查询并更新角色权限。
5. 同账号新登录可使旧令牌失效。
6. 不同账号并发登录互不影响。
7. 管理员可维护预警规则并即时生效。
8. 医生可查看患者洞察数据。
9. 医生预警列表支持风险分过滤与排序。
10. 医生仅能处理其团队范围内患者预警。
11. 管理员禁用指标后，患者上报被正确拦截。
12. 连续高风险和趋势上升可触发主动预警。
13. 个性化阈值可覆盖默认规则并支持启停回退。

### 6.2.2 自动化测试结果

自动化脚本执行结果表明，核心接口链路可稳定返回预期状态码与数据结构：

1. 接口回归脚本覆盖场景 1 到 13，支持跨平台执行。
2. 错误码门禁脚本可独立验证 400、401、403、404、409。
3. 脚本断言失败时返回非 0 退出码，便于 CI 拦截。

### 6.2.3 单元与集成测试结果

1. 单元测试覆盖用户管理、健康数据校验、预警判定等核心服务逻辑。
2. 集成测试覆盖 Flyway 迁移回归与安全错误码链路。
3. Flyway 测试在无 Docker 环境下可自动跳过，保证环境兼容性。

### 6.2.4 前端功能验证结果

1. 登录鉴权流程正常，token 失效后能正确跳转登录页。
2. 患者、医生、管理员页面关键操作均可完成并返回结果提示。
3. 重点图表页面可正常渲染，筛选和视图切换交互有效。

## 6.3 性能测试

### 6.3.1 性能测试目标

考虑课程设计场景，本系统性能测试以“可用性验证与瓶颈识别”为目标，而非超大规模极限压测。重点关注：

1. 认证接口在连续访问下的响应稳定性。
2. 预警列表与历史数据分页接口在常规并发下的可响应性。
3. 复杂聚合接口在多条件查询时的可接受时延。

### 6.3.2 性能测试对象与指标

测试对象选择用户实际高频接口：

1. 登录接口：POST /api/auth/login。
2. 患者历史查询接口：GET /api/patient/data。
3. 医生预警查询接口：GET /api/doctor/alerts。
4. 管理日志分页接口：GET /api/admin/logs/page。

观察指标包括：

1. 接口成功率。
2. 平均响应时间。
3. P95 响应时间。
4. 错误率与超时率。

### 6.3.3 性能测试方法

在现有工程中采用轻量压测方法：

1. 以脚本方式持续调用高频接口，模拟并发访问。
2. 配合系统监控页观察总请求量与关键业务指标波动。
3. 对异常响应进行日志回溯，定位慢查询或参数异常。

### 6.3.4 性能测试结果

结合接口回归与轻量压力运行结果，可得到以下结论：

1. 核心接口在测试并发范围内成功率保持稳定。
2. 未出现大面积超时、线程阻塞或服务不可用现象。
3. 影响响应性能的主要因素集中在查询条件复杂度与数据量增长。
4. 通过分页约束、缓存策略与索引查询，系统可满足课题规模下的性能要求。

### 6.3.5 性能优化措施

系统已实现或采用以下优化策略：

1. 分页参数边界保护，避免超大 pageSize 导致单次查询过重。
2. 高频读接口缓存化，减少重复查询压力。
3. 写操作后定向清理缓存，平衡一致性与性能。
4. 关键列表按时间倒序并结合索引字段查询，降低扫描成本。
5. 日志导出设置上限，避免一次导出造成资源占用峰值。

## 6.4 结果分析

### 6.4.1 功能正确性分析

综合自动化与手工测试，系统功能正确性达到预期：

1. 三类角色主流程均可闭环执行。
2. 权限边界与数据隔离机制有效，越权请求可被拦截。
3. 规则配置、个性化阈值与趋势预警联动正确。

### 6.4.2 稳定性与可维护性分析

1. 通过错误码门禁与 CI 质量门禁，回归风险可前置发现。
2. 测试脚本支持跨平台执行，便于本地复现与持续集成。
3. 操作日志与反馈链路完善，问题定位与追溯成本较低。

### 6.4.3 性能分析

1. 系统在课程设计目标负载下具备稳定响应能力。
2. 当前性能瓶颈主要来自数据规模扩大后的查询聚合开销。
3. 已有缓存与分页机制可有效延缓性能退化趋势。

### 6.4.4 存在问题与改进方向

1. 目前性能测试以轻量压测为主，后续可引入标准压测工具进行更系统的并发评估。
2. 可进一步补充接口级性能基线与版本对比报告，形成量化优化闭环。
3. 可将前端关键交互纳入自动化 UI 回归，减少人工回归成本。

## 本章小结

本章通过测试用例设计、功能验证、性能评估和结果分析，证明系统已具备较完整的可用性与稳定性。测试结果显示，系统能够满足院外健康管理场景的业务需求，并在安全、权限、规则联动和审计追溯方面具有较好的工程落地能力，为后续迭代优化提供了可靠基础。
